🔴 BUG 1: AUTENTICAÇÃO FALTANTE [CRÍTICO]

Arquivo:        backend/src/routes/cliente.ts
Status:         ✅ CONFIRMADO
Severidade:     🔴🔴🔴 CRÍTICA
Impacto:        MÁXIMO

Descrição:
  Todas as 25+ rotas de cliente são publicamente acessíveis sem autenticação.
  Qualquer pessoa consegue listar/modificar documentos de qualquer cliente.

Exemplo:
  # Sem autenticação, atacante consegue:
  curl http://localhost:3000/api/cliente/bob-uuid/documentos
  curl -X PATCH http://localhost:3000/api/cliente/documento/doc-id/status \
    -d '{"status":"REJECTED"}'

Causa Raiz:
  - Rotas cliente não têm middleware de autenticação
  - Diferente das rotas juridico que têm authMiddleware
  - Controllers não validam req.user

Afetados:
  ✗ GET /cliente/:clienteId/documentos
  ✗ PATCH /documento/:documentoId/status
  ✗ POST /uploadDoc
  ✗ DELETE /documento/:documentoId
  ✗ GET /cliente/:clienteId/notificacoes
  ✗ GET /cliente/:clienteId/processos
  ... (20+ endpoints mais)

🟡 BUG 2: AUDITORIA SEM VALIDAÇÃO [IMPORTANTE]

Arquivo:        backend/src/controllers/cliente/ClienteDocumentController.ts
Status:         ✅ CONFIRMADO
Severidade:     🟡🟡 IMPORTANTE
Impacto:        MÉDIO-ALTO

Descrição:
  Campo 'analisado_por' aceita qualquer UUID de req.body sem validação.
  Qualquer um consegue falsificar análises de documentos.

Exemplo:
  # Cliente consegue fazer isso:
  PATCH /api/cliente/documento/doc-123/status
  {
    "status": "REJECTED",
    "analisadoPor": "admin-uuid-fake"  // ❌ UUID inventado!
  }
  
  Resultado: Banco registra que admin-fake rejeitou o documento.

Código Problemático:
  const { status, motivoRejeicao, analisadoPor } = req.body
  // ❌ Confiou cegamente no cliente

Causa Raiz:
  - Aceita analisadoPor de req.body em vez de req.user
  - Sem middleware que passa usuário autenticado
  - Sem validação de role/permissão

Impacto:
  ✗ Auditoria falsificada
  ✗ Impossível rastrear análises reais
  ✗ Não-compliance com requisitos legais
